CUDA Profiling Day 1

发表于 2018-11-25

nvprof

使用nvprof [options] [application] [application-args]命令来profile cuda程序。

  • 使用--metrics [metric-name,...]选项来指定需要检测的项。
    • 使用--query-metrics选项来查看所有available的metrics。
  • 使用--analysis-metrics选项选择适配Visual Profiler的metrics。

nvvp (Visual Profiler)

使用姿势：在服务器上用byobu+nvprof --analysis-metrics -o xx.nvvp ...跑程序，跑完scp回本地用nvvp可视化profile结果。

  • 名义上支持导出结果到PDF。我测试了一下，抛出了java.lang.reflect.InvocationTargetException。
  • 支持Source mapping，但要在nvcc的编译选项里加-lineinfo。

我profile的结果中stall主要由于global memory和inactive instructions组成，想一想基本合理，branch指令比较多（这里滚回去学习了一下cuda SIMT相关的基础知识，摔）

另外，nvcc总是为每个thread分配maxRegistersPerThread数量的寄存器，结果是SM分配到maxThreadsPerMultiprocessor一半的线程时就用掉了SM中全部的寄存器，GPU utilization永远不到50%。这里其实可以使用-maxrregcount或__launch_bounds__在寄存器数和线程数之间做出一些权衡，但要考虑具体的硬件数值。我在一块Tesla K40上将registerPerThread减半以加倍SM线程数，内核性能提升比较明显（2.07s -> 1.68s）。

Reference

  • cuda documentation
  • Shane Cook - CUDA Programming
  • many references
  • https://stackoverflow.com/questions/17553417/cuda-registers-per-thread
  • https://stackoverflow.com/questions/44704506/limiting-register-usage-in-cuda-launch-bounds-vs-maxrregcount

Template Based Component Multiplexing

前言

我喜欢用C++模板做DSL。

工厂模式

使用模板DSL的劣势是需要configuration需要在编译时确定，对于实际应用而言这是不可容忍的。我们希望能通过运行时的配置来生成需要实例化的类。

事实上，为了实例化所有的工厂类型T<>，只要在编译时对所有合法的configuration进行排列组合即可。我们可以为工厂传入合法的configuration，然后在工厂的构造函数中进行排列组合，再从中筛选出所有合法的枚举并注册它们。
